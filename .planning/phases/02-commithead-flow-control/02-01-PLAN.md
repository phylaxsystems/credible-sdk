<objective>
Lay the contractual foundation for Phase 2: refactor `StateWorker::process_block` so it returns `BlockStateUpdate` instead of writing to MDBX directly, define the `CommitHeadSignal` channel type, add `StateWorkerConfig` to sidecar config, and extend `StateWorkerError` with the variants needed by the buffer/flush logic. All downstream plans in this phase depend on these contracts being stable.

Purpose: The Phase 1 scaffold has a no-op `run_blocking_inner`. Phase 2 replaces it with a real buffer+flush loop (Plan 03) and wires the CommitHead channel to CoreEngine (Plan 02). Both of those plans need: (1) a `process_block` that returns `BlockStateUpdate` so the buffer can accumulate updates without immediately committing, (2) a `CommitHeadSignal` type to carry block numbers over the flume channel, (3) a `StateWorkerConfig` so `StateWorkerThread` knows the WS URL, MDBX write path, genesis path, and state depth at construction time, and (4) error variants covering MDBX write failures and tracing errors.

Output: Modified `worker.rs` (process_block returns BlockStateUpdate), adapted `state-worker/src/main.rs` (calls commit_block on returned update), new `CommitHeadSignal` type in `state_worker_thread/mod.rs`, `StateWorkerConfig` struct in `args/mod.rs`, extended `StateWorkerError` in `error.rs`.
</objective>

/// Signal sent by CoreEngine to StateWorkerThread after each committed block.
///
/// Carries only the block number — the state worker uses this to flush all
